# System Requirements Specification: MAGI (Multi-modal Autonomous General Intelligence) Decision System

## 1. Introduction

### 1.1. Purpose

The purpose of this document is to define the functional and non-functional requirements for the MAGI Decision System. The system is a simplified implementation of the fictional MAGI supercomputer from *Neon Genesis Evangelion*, simulating a consensus-based decision-making process between three distinct persona-based nodes (Melchior-1, Balthasar-2, Casper-3) to resolve critical queries, particularly "Self-Destruct" motions. The system will feature a graphical user interface (GUI) modeled after the NERV console as depicted in `image_0.png`, `image_1.png`, and `image_2.png`.

### 1.2. Scope

The system will accept critical motions via text-based natural language queries. It will simulate a deliberation and voting process where Melchior, Balthasar, and Casper generate a vote based on a pre-defined random-simulated persona logic. The GUI will display the status, query, deliberation results, and the individual votes of each node, complete with the specific visual aesthetics of the original anime. The final decision requires a unanimous vote. The system should be responsive and compatible with both desktop and mobile web views.

## 2. General Description

### 2.1. System Context

The system operates as a standalone web application. A User interacts with the MAGI system via a text input and views the results on the emulated NERV interface.

[User] -> [Text Query] -> [**MAGI System**] -> [Decision Output] & [GUI Display]

### 2.2. User Persona

The primary user is an operator at a fictional NERV headquarters.

### 2.3. Principles of Operation

The core principle is to emulate a consensus-based, persona-driven artificial intelligence. Each node (Melchior-1, Balthasar-2, Casper-3) is designed with a unique decision-making bias (e.g., Scientist, Mother, Woman), and a unanimous vote is required for critical motions like "Self-Destruct." The system will use current LLM principles (Large Language Models) to parse and understand natural language, but for this version, the actual votes are generated by a random simulation function that mimics those biased personas.

## 3. Technical Requirements

### 3.1. General Technology Stack

* **Language:** Python
* **Web Framework:** Flask (recommended for its simplicity and Python compatibility)
* **GUI / Styling:** HTML5, CSS3, JavaScript
* **Responsive Framework:** Bootstrap (recommended for mobile-first, responsive design)
* **Persona Simulation:** Python `random` module with custom probability matrices.

### 3.2. Detailed GUI Styling and Fonts (Anime Emulation)

The GUI must be a high-fidelity recreation of the NERV console.

* **Color Palette:**
* Background: Dark grey with a faint grid/scanline pattern (e.g., `#333333` with an overlay).
* Node Panels: Specific color codes as referenced in `image_0.png` (Orange-red `#FF5733`, Blue `#3399FF`, Light Grey `#CCCCCC`).
* Text: Bright Orange-Red (for status, labels, and critical text), Bright Red (for '否決' / 'DISSENT' and critical motions), and bright Green/Yellow-Green (for '承認' / 'APPROVE' and other text).
* Diagonal Hazards: Alternating thick red and grey diagonal stripes (as seen in `image_0.png`).


* **Typography:** The font must be a highly specialized, clean, high-tech-looking, geometric sans-serif font that closely mimics the one used in the anime. A font like 'Arial' or 'Helvetica' is insufficient.
* **Responsiveness:** The CSS must use media queries (`@media`) to adjust the layout and font sizes for mobile screens, switching from a three-column layout (as in `image_0.png` and `image_2.png`) to a single-column stacked layout suitable for narrow portrait views. The text must remain legible at smaller sizes. The `pixiv.net` watermarking should not be part of the UI.

## 4. Functional Requirements

### 4.1. Text Query Input

* **FR-QI-01:** The system shall provide a text input field for users to enter natural language queries.
* **FR-QI-02:** The input field shall be styled to match the NERV interface.
* **FR-QI-03:** The system shall accept a query upon user pressing `Enter`.

### 4.2. Motion Detection & Parsing

* **FR-MDP-01:** The system shall use simple LLM-inspired pattern matching or basic NLP (e.g., string matching for this version) to detect critical "Self-Destruct" motions.
* Example: A query like "Initiate self-destruct" should trigger the critical workflow.


* **FR-MDP-02:** When a "Self-Destruct" motion is detected, the status text shall change to "MOTION: SelfDestruct" as seen in `image_0.png`. The red-grey striped areas should activate. The Access Mode should be "SuperUser".

### 4.3. Persona-Based Voting Simulation

* **FR-PBV-01:** The system shall simulate three separate voting processes, one for each node:
* **MELCHIOR-1 (Scientist):** Simulates a logic-driven decision.
* **BALTHASAR-2 (Mother):** Simulates an ethics-driven decision.
* **CASPER-3 (Woman):** Simulates an intuition-driven decision.


* **FR-PBV-02:** When a "Self-Destruct" motion is detected, each node shall generate a random vote weighted according to its simulated persona.
* **CASPER-3 (Woman):** Shall have a 100% (or very high) probability of voting 'DISSENT' (否決) on all "Self-Destruct" motions, as per the source material.
* **MELCHIOR-1** and **BALTHASAR-2**: Shall have varying high probabilities (e.g., 90%) to vote 'APPROVE' (承認) on "Self-Destruct" motions under specific conditions, which can be randomly simulated. For this version, they will always vote `APPROVE` to guarantee the unique Casper-driven dissent condition.



### 4.4. Decision Execution & Display

* **FR-DED-01:** The system shall require a unanimous vote (all three nodes `APPROVE`) for a "Self-Destruct" motion to pass.
* **FR-DED-02:** The GUI shall update dynamically (potentially with a small deliberate delay to simulate calculation) to show:
* The overall status "RESULT OF THE DELIBERATION".
* A central block with the text "MAGI TOKYO-03".
* Three large blocks (one per node), positioned as in `image_0.png`, containing the node's name, number (e.g., CASPER-3), and its final vote.


* **FR-DED-03:** The vote text inside each node block must use the specific characters:
* '承認' (APPROVE) for affirmative votes (e.g., Melchior and Balthasar).
* '否決' (DISSENT) for negative votes (e.g., Casper).


* **FR-DED-04:** The text color and block layout should match the visual reference of `image_0.png`.
* **FR-DED-05:** Upon a non-unanimous vote, the final system state should display "DISSENT" and the motion fails.
* **FR-DED-06:** Secondary text, like "Direct Link Connection", "Access Mode:SuperUser", and the specific code `CODE:258`, `FILE: MAGI.SYS`, etc., from `image_1.png` should be included on the display as decorative but visually accurate elements. The "提訴: 自律自爆" (MOTION: Self-Destruct) should also be present.

## 5. Non-Functional Requirements

### 5.1. Performance

The system should provide a responsive user interface with low latency between input and output. Simulated "calculation" delays should be aesthetic and not impact usability.

### 5.2. Reliability

The system should consistently parse the query, run the simulation, and update the GUI as defined.

### 5.3. Usability

The interface should be clear, despite its specialized styling. The input field and result display must be easily identifiable on both desktop and mobile platforms.

### 5.4. Cross-Platform Compatibility

The web application must be tested and confirmed to render and function correctly on major desktop browsers (Chrome, Firefox, Safari, Edge) and major mobile browsers (iOS Safari, Android Chrome).

## 6. Implementation Notes (Python/Flask Example Structure)

The core logic could be structured as follows:

```python
import random

def simulate_magi_voting(query):
    is_self_destruct = "self-destruct" in query.lower()

    if is_self_destruct:
        # Core narrative logic: Casper-3 always dissents to self-destruct.
        # Melchior-1 and Balthasar-2 approve.
        votes = {
            "MELCHIOR-1": "承認",  # APPROVE
            "BALTHASAR-2": "承認",  # APPROVE
            "CASPER-3": "否決",    # DISSENT
        }
    else:
        # For non-self-destruct queries, simulate other persona-weighted logic.
        # This could be random for now.
        votes = {
            "MELCHIOR-1": "承認" if random.random() > 0.1 else "否決",
            "BALTHASAR-2": "承認" if random.random() > 0.2 else "否決",
            "CASPER-3": "承認" if random.random() > 0.3 else "否決",
        }

    # Unanimous decision check
    unanimous_approved = all(v == "承認" for v in votes.values())
    unanimous_dissented = all(v == "否決" for v in votes.values())

    if unanimous_approved:
        final_decision = "APPROVED"
    elif unanimous_dissented:
        final_decision = "DISSENTED"
    else:
        final_decision = "REJECTED (Split Vote)"

    return votes, final_decision

# Ex: 'votes, final_decision = simulate_magi_voting("Initiate self-destruct sequence.")'

```

This logic would be integrated into a Flask application to handle user input from the front end and return the vote data for rendering on the high-fidelity GUI.
